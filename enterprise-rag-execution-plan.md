# 企业级 RAG 知识库执行方案

## 1. 文档目标

本方案用于指导当前 `python-ai-agent` 项目从“单体上传检索 Demo”升级为“企业级组织化知识平台”。

目标不是只完成文档上传和问答，而是构建一套可长期演进的系统：

- 面向企业组织结构，而不是单用户个人知识库
- 支持部门、上下级组织、共享范围和审批流
- 支持权限过滤后的知识检索，而不是全库混查
- 支持审计、治理、版本、发布、下架等企业能力
- 支持前端页面与后端接口联调，形成完整业务闭环

本方案强调：

- 先搭好领域模型，再写功能
- 先做“权限可控”，再做“回答更聪明”
- 先做“可治理的知识空间”，再做“无限制的全局搜索”

---

## 2. 当前项目现状

当前项目已具备以下基础能力：

- `FastAPI` 后端服务
- 用户注册 / 登录 / 获取当前用户信息
- 文档上传、解析、切片、向量化、Qdrant 入库
- 基础 RAG 检索链路
- OpenAI 兼容模型接入，支持 `qwen-plus` 与 `text-embedding-v3`
- PostgreSQL + Redis + Qdrant 基础设施

当前不足：

- 用户没有组织归属模型
- 没有部门、角色、成员关系、审批流程
- 文档没有明确的业务归属与可见性模型
- 检索没有严格按权限范围过滤
- 没有知识库空间概念
- 没有文档状态、共享策略、审计机制
- 前端尚未形成正式企业级工作台与管理台

因此，接下来的建设重点不应该只是“加几个接口”，而应该先完成平台化抽象。

---

## 3. 总体设计原则

### 3.1 不是只做 RBAC，而是做组合权限模型

单纯 RBAC 不足以支撑企业知识治理。推荐采用：

- `RBAC`：定义用户能执行哪些动作
- `组织范围控制`：定义用户可作用到哪些部门
- `资源策略控制`：定义文档/知识库可被哪些人查看

最终权限判断应基于：

`动作权限 + 组织范围 + 资源可见性`

### 3.2 不是“文档中心”，而是“知识空间中心”

推荐核心关系：

`用户 -> 部门成员关系 -> 知识库空间 -> 文档 -> 切片 -> 向量`

原因：

- 企业里一个部门通常有多个知识空间
- 文档需要归属于空间，而不是直接归属于人
- 空间是权限、共享、审批、统计的核心载体

### 3.3 权限过滤必须在检索链路里生效

不能先在全量向量库召回，再做后过滤。

应改为：

1. 先计算当前用户可访问的知识空间范围
2. 再计算可访问文档范围
3. 再在向量检索时带上过滤条件
4. 最后返回可见内容及引用来源

这是企业级 RAG 的安全底线。

### 3.4 前端不是“接口调试页”，而是“业务工作台”

前端应承担以下作用：

- 组织身份入口
- 知识空间入口
- 文档治理入口
- 审批待办入口
- 管理与审计入口
- 权限内智能问答入口

## 4. 目标业务模型

### 4.1 用户与组织

推荐实体：

- `users`
- `departments`
- `department_memberships`
- `department_join_requests`

关键建议：

- 不要把 `department_id` 直接塞到 `users`
- 一个用户可能有多个部门关系
- 需要支持主部门、协作部门、借调关系、历史状态

推荐关系：

- `users`：账号主体
- `departments`：组织节点，支持父子结构
- `department_memberships`：成员关系表
- `department_join_requests`：入部门申请表

### 4.2 角色与权限

推荐实体：

- `roles`
- `user_roles`
- `role_permissions`

建议角色分层：

- 平台角色
  - `platform_admin`
  - `security_admin`
  - `auditor`
- 组织角色
  - `department_director`
  - `department_manager`
  - `department_member`
- 知识角色
  - `knowledge_admin`
  - `knowledge_operator`
  - `knowledge_reviewer`

说明：

- 平台角色控制全局能力
- 组织角色控制部门治理
- 知识角色控制文档与知识库管理

### 4.3 知识库空间

推荐实体：

- `knowledge_bases`

每个知识库空间应具备：

- 所属部门
- 名称与描述
- 状态
- 默认可见性策略
- 是否允许上传
- 是否需要文档审批
- 是否允许跨部门共享

示例：

- 人事制度库
- 研发规范库
- 销售资料库
- 项目交付知识库

### 4.4 文档与版本

推荐实体：

- `documents`
- `document_versions`
- `document_chunks`

建议不要把一个文件只看成“一条记录”，而是分层：

- `documents`：业务文档实体
- `document_versions`：版本记录
- `document_chunks`：切片记录

推荐字段方向：

- 文档标题
- 上传人
- 所属知识库
- 所属部门
- 当前状态
- 可见性策略
- 是否启用检索
- 当前版本号
- 文件元信息

### 4.5 共享与审批

推荐实体：

- `document_shares`
- `approval_requests`
- `approval_records`

文档共享不建议只用 `is_shared` 布尔值，应支持：

- 仅本部门可见
- 本部门及下级部门可见
- 指定部门可见
- 全公司可见
- 限定角色可见

审批流至少包含：

- 加入部门审批
- 文档发布审批
- 文档共享审批
- 权限升级审批

### 4.6 审计与治理

推荐实体：

- `audit_logs`
- `search_logs`
- `document_access_logs`

建议记录：

- 谁上传了什么
- 谁审批了什么
- 谁查询了什么
- 哪次问答命中了哪些文档
- 谁访问了敏感资料

## 5. 推荐权限模型

## 5.1 权限判断结构

权限判断建议拆成三层：

### 第一层：动作权限

例如：

- 是否允许上传文档
- 是否允许审批申请
- 是否允许查看审计日志
- 是否允许创建知识库

### 第二层：组织范围

例如：

- 仅自己部门
- 自己部门 + 下级部门
- 全公司

### 第三层：资源可见性

例如：

- 文档属于本部门
- 文档被共享给当前部门
- 文档对全公司可见
- 文档仅上传者和管理员可见

最终建议统一为权限服务判断，而不是散落在各 API 中临时判断。

## 5.2 上级查看下属知识库

建议基于部门树实现，而不是基于角色名硬编码。

部门表建议支持：

- `parent_id`
- `path` 或闭包表

这样可以统一支持：

- 上级查看下属
- 上级审批下属加入申请
- 上级汇总统计
- 上级统一治理知识空间

## 5.3 注册后的默认权限

推荐新用户默认状态：

- 可登录
- 无部门归属
- 无知识库访问权限
- 仅可访问个人页、申请页、公共帮助页

审批通过加入部门后，才获得正式业务权限。

---

## 6. 文档可见性设计

推荐至少支持以下可见性级别：

- `private_to_department`
  - 仅本部门成员可见
- `shared_to_descendants`
  - 本部门及下级部门可见
- `shared_to_specific_departments`
  - 指定部门可见
- `org_public`
  - 全公司登录用户可见
- `restricted`
  - 仅上传者、审批者、管理员或明确授权人可见

后续如涉及敏感资料，可扩展：

- 禁止进入通用问答
- 仅允许精确检索
- 禁止下载
- 页面加水印
- 访问强审计

---

## 7. RAG 检索设计

### 7.1 当前问题

当前项目的检索模型更偏 Demo：

- 所有文档统一进入一个集合
- 文档 payload 缺少完整权限维度
- 检索前没有严格做用户可见范围过滤

### 7.2 推荐目标

每个 chunk payload 至少包含：

- `knowledge_base_id`
- `department_id`
- `document_id`
- `document_version_id`
- `visibility_level`
- `shared_department_ids`
- `status`
- `owner_user_id`

### 7.3 推荐检索链路

1. 用户输入问题
2. 根据用户身份计算允许访问的知识空间
3. 根据知识空间和共享策略计算允许访问的文档
4. 检索时带 payload filter 到 Qdrant
5. 返回结果时附带来源信息
6. 大模型生成答案时展示引用依据

### 7.4 推荐回答展示

问答结果必须可追溯，前端应展示：

- 答案内容
- 命中的知识库
- 命中的部门
- 命中的文档名称
- 命中的文档片段
- 相似度或相关度

企业场景下，可信度比“看起来聪明”更重要。

## 8. 前端页面规划

建议前端采用“工作台 + 管理台 + 问答台”三层结构。

### 8.1 认证与组织入口

页面建议：

- 登录页
- 注册页
- 首次登录补充信息页
- 申请加入部门页
- 查看申请状态页

### 8.2 工作台

页面建议：

- 我的部门
- 我的知识库
- 最近访问文档
- 最近上传记录
- 我的审批待办
- 常用问答入口

### 8.3 知识库页面

页面建议：

- 部门树浏览
- 知识库空间卡片
- 文档列表
- 筛选与搜索
- 可见范围标识
- 共享状态标识

### 8.4 文档治理页面

页面建议：

- 文档上传
- 文档版本
- 文档状态流转
- 共享设置
- 下架/归档操作
- 切片与命中预览

### 8.5 审批中心

页面建议：

- 部门加入审批
- 文档发布审批
- 文档共享审批
- 权限升级审批

### 8.6 管理后台

页面建议：

- 部门管理
- 角色管理
- 知识库空间管理
- 系统配置
- 审计与统计

### 8.7 智能问答页

问答页必须支持：

- 选择检索范围
- 仅搜索自己可访问空间
- 展示命中来源
- 查看引用文档片段
- 追问与上下文保留

## 9. 后端接口分组规划

建议未来按领域拆分 API：

- `auth`
  - 注册、登录、当前用户
- `users`
  - 个人信息、成员身份、角色信息
- `departments`
  - 部门树、部门详情、成员列表
- `department-requests`
  - 加入申请、审批、审批历史
- `roles`
  - 角色与权限管理
- `knowledge-bases`
  - 空间管理、可见范围、成员授权
- `documents`
  - 上传、列表、详情、版本、下架、归档
- `document-shares`
  - 共享策略与共享审批
- `rag`
  - 权限内检索、问答、引用返回
- `audit`
  - 审计日志、访问日志、检索日志
- `admin`
  - 平台级治理接口

建议原则：

- “业务对象”与“调试接口”分开
- 审批类接口单独建域
- 问答接口与文档管理接口分离

## 11. 推荐实施阶段

## 阶段一：组织基础设施

目标：

- 完成部门树
- 完成成员关系
- 完成加入部门申请与审批
- 完成基础角色模型

交付结果：

- 新用户注册后可发起部门加入申请
- 部门负责人可审批
- 审批通过后用户获得组织身份

## 阶段二：知识库空间

目标：

- 引入 knowledge base 概念
- 知识库按部门归属
- 用户只看到自己有权限的知识库

交付结果：

- 部门负责人可创建知识库
- 成员按权限进入对应空间

## 阶段三：文档治理

目标：

- 文档归属知识库
- 文档具备状态、版本、可见性
- 上传与共享可控

交付结果：

- 支持文档私有 / 指定共享 / 组织公开
- 支持文档发布与下架

## 阶段四：权限过滤 RAG

目标：

- 检索严格按权限过滤
- payload 带齐组织和共享字段
- 前端展示引用来源

交付结果：

- 用户只能问到自己有权限看到的内容
- 回答具备可追溯出处

## 阶段五：审批与审计

目标：

- 加入审批、共享审批、发布审批形成闭环
- 审计日志与访问日志可追踪

交付结果：

- 系统具备企业治理能力

## 12. MVP 建议

如果要尽快落地，推荐最小可用版本包含：

- 注册 / 登录
- 部门树
- 加入部门申请
- 部门负责人审批
- 知识库空间
- 文档上传
- 文档可见性：本部门 / 指定部门 / 全公司
- 权限过滤检索
- 上级查看下属部门知识库
- 基础审计日志

先不做：

- 复杂流程编排引擎
- 字段级脱敏
- 高级密级管控
- 多租户隔离
- 全量运营报表

这样可以更快形成真正可用的企业级第一版。

## 13. 风险与关键注意事项

### 13.1 不要把完整权限树塞进 JWT

建议 JWT 只存：

- user_id
- 基础身份
- 会话信息

权限应由后端按数据库或缓存实时判断。

### 13.2 不要只做布尔共享

共享必须是策略模型，而不是 `is_shared = true/false`。

### 13.3 不要让全部文档进入无过滤混合检索

必须保证权限范围先于召回。

### 13.4 不要跳过文档状态机

建议至少有：

- draft
- pending_review
- published
- archived
- disabled

### 13.5 不要忽略审计能力

企业知识库一旦进入正式环境，审计不是可选项。

---

## 14. 对当前项目的最优演进结论

如果以“企业级 RAG 知识库”为目标，当前项目的最优路线不是简单叠加几个接口，而是：

1. 建立组织模型
2. 建立成员与审批模型
3. 建立知识库空间模型
4. 建立文档治理模型
5. 建立策略化共享模型
6. 建立权限过滤 RAG
7. 建立审计与治理能力
8. 最后再做更强的问答体验与运营能力

最重要的一句话：

`先把知识治理做对，再把智能问答做强。`
