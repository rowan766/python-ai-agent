# 企业级 RAG 前端页面信息架构草案

## 1. 文档目标

本文件承接以下三份方案文档：

- 《企业级 RAG 知识库执行方案》
- 《企业级 RAG 数据库表设计草案》
- 《企业级 RAG 前后端接口设计草案》

目标：

- 规划企业级 RAG 平台前端页面结构
- 明确每个页面服务的业务目的
- 明确普通成员、负责人、管理员分别看到什么
- 明确主要页面间跳转关系
- 明确每个页面需要承载的核心模块

本文件只做页面和信息架构设计，不写具体实现代码。

---

## 2. 产品定位

这个前端不应只是“接口调用页面”，而应是一个企业级知识工作台。

核心定位：

- 用户进入系统后，先获得组织身份
- 再进入有权限的知识空间
- 在知识空间中完成上传、治理、共享、审批、问答
- 所有问答结果都能回到来源文档与组织归属

因此前端信息架构应服务四个核心目标：

- 身份与组织归属
- 知识空间管理
- 文档治理与审批
- 权限内检索与问答

## 3. 页面结构总览

建议采用三层式产品结构：

### 第一层：进入层

- 登录
- 注册
- 首次登录引导
- 申请加入部门
- 等待审批

### 第二层：工作层

- 工作台
- 知识库首页
- 文档中心
- 问答页
- 审批中心

### 第三层：治理层

- 部门管理
- 角色管理
- 知识库配置
- 审计中心
- 系统配置

建议导航结构：

- 顶部：全局搜索、当前部门、个人信息、通知
- 左侧：主导航
- 主内容区：页面业务内容
- 右侧：上下文面板或引用来源面板

## 5. 页面地图建议

## 5.1 认证与组织入口

页面建议：

- `/login`
- `/register`
- `/onboarding`
- `/join-department`
- `/join-department/status`

### 登录页

目标：

- 完成认证

核心模块：

- 邮箱/用户名登录
- 密码输入
- 记住登录态
- 忘记密码入口

登录后跳转逻辑：

- 有有效组织身份：进入工作台
- 无组织身份：进入 onboarding

### 注册页

目标：

- 完成账号创建

核心模块：

- 邮箱
- 用户名
- 密码
- 注册说明

注册成功后：

- 自动登录
- 进入 onboarding

### Onboarding 页面

目标：

- 引导新用户补足企业使用入口

核心模块：

- 个人信息补充
- 当前状态提示
- 申请加入部门入口
- 查看组织说明

### 申请加入部门页

目标：

- 发起加入组织申请

核心模块：

- 部门树选择器
- 申请身份选择
- 申请理由
- 提交按钮

### 申请状态页

目标：

- 告知用户是否已通过审批

核心模块：

- 当前申请状态
- 审批意见
- 是否允许撤回 / 重新申请

---

## 5.2 工作台

页面建议：

- `/workspace`

目标：

- 成为用户进入系统后的默认首页

核心模块：

- 当前组织身份卡片
- 我的知识库
- 最近访问文档
- 最近上传
- 最近检索
- 待审批事项
- 快速问答入口

建议布局：

- 顶部：欢迎区 + 当前部门切换
- 中部：知识库卡片 + 最近记录
- 侧边：待办事项 + 审批数 + 系统通知

工作台的意义：

- 不让用户一进来就面对复杂管理界面
- 先从“我相关的东西”开始

- 活动

概览模块建议：

- 空间简介
- 文档数量
- 可见范围说明
- 最近更新

## 5.4 文档中心

页面建议：

- `/documents`
- `/documents/:id`
- `/documents/:id/versions`
- `/documents/:id/shares`

### 文档列表页

目标：

- 管理当前用户可访问或可治理的文档

核心模块：

- 知识库筛选
- 部门筛选
- 状态筛选
- 可见性筛选
- 关键字搜索
- 批量操作

每条文档建议展示：

- 标题
- 所属知识库
- 所属部门
- 上传人
- 状态
- 可见性
- 最近更新时间

### 文档详情页

目标：

- 成为文档治理核心页面

建议分区域：

- 顶部：标题、状态、操作按钮
- 左侧：文档元信息
- 中间：文档详情 / 版本 / 切片预览
- 右侧：可见性、共享、审批信息

核心模块：

- 文档概览
- 状态流转
- 当前版本信息
- 引用命中预览
- 切片预览
- 共享策略摘要
- 审批记录摘要

### 文档版本页

目标：

- 展示版本历史与回滚语义

核心模块：

- 版本时间线
- 变更说明
- 上传人
- 当前版本标记
- 是否已进入检索

### 文档共享页

目标：

- 配置共享边界

核心模块：

- 当前共享策略列表
- 新增共享策略
- 指定部门选择器

## 5.5 上传页

页面建议：

- `/documents/upload`

目标：

- 完成知识库归属型上传，而不是无上下文上传

核心模块：

- 选择知识库空间
- 上传文件
- 文档标题
- 文档描述
- 标签
- 可见性设置
- 共享设置
- 提交上传

建议上传完成后的反馈区展示：

- 上传成功状态
- 文档 ID
- 当前版本号
- 切片数量
- 是否进入审批
- 是否进入检索

说明：

- 企业级产品里，上传页必须先选“归属到哪个知识库”
- 不建议继续保留纯 `/rag/upload` 那种没有业务归属的体验

## 7. 页面之间的关键跳转关系

建议核心跳转链路如下：

### 链路一：新用户入场

登录/注册
-> Onboarding
-> 申请加入部门
-> 等待审批
-> 工作台

### 链路二：知识治理

工作台
-> 知识库列表
-> 知识库详情
-> 文档列表
-> 文档详情
-> 文档共享 / 版本 / 发布

### 链路三：审批

通知/工作台待办
-> 审批中心
-> 审批详情
-> 审批动作
-> 返回工作台或资源详情

### 链路四：问答

工作台 / 知识库详情
-> 问答页
-> 引用来源
-> 文档详情

### 链路五：治理与审计

管理后台
-> 部门管理 / 角色管理 / 审计中心

## 8. 前端状态管理建议

建议至少区分以下状态域：

- 登录态
- 当前用户资料
- 当前部门上下文
- 当前可访问知识库范围
- 待审批数量
- 当前问答会话

前端应避免：

- 把完整权限逻辑硬编码在页面里
- 假设前端能决定最终可访问范围

前端更适合负责：

- 展示状态
- 发送用户意图
- 渲染后端返回的有效范围

后端负责：

- 真正权限判断
- 有效范围裁剪
- 最终结果控制

## 9. 页面设计原则建议

### 9.1 信息可信优先于炫技

企业级问答页最重要的不是“看起来智能”，而是：

- 有来源
- 能追溯
- 可验证

### 9.2 不让权限成为隐形逻辑

建议页面显式展示：

- 你当前所属部门
- 你当前可访问的知识库范围
- 这篇文档为什么你能看
- 这次问答在哪些知识库中检索

### 9.3 不把治理操作埋得太深

文档状态、共享设置、审批信息，应在文档详情页明显可见。

### 9.4 区分“浏览者界面”和“管理者界面”

避免所有用户都看到复杂治理控件。

建议根据角色动态呈现页面模块。

## 11. 设计结论

企业级 RAG 前端不应只是一个“问答框 + 上传按钮”的页面，而应是一个围绕组织、知识空间、文档治理、审批和审计构建的工作平台。

最优前端信息架构应满足：

- 进入有门槛
- 组织身份清晰
- 知识空间可导航
- 文档治理可操作
- 权限范围可感知
- 问答结果可追溯
- 管理能力可收敛

一句话总结：

`企业级 RAG 前端的核心不是聊天，而是可信、可控、可治理的知识工作流。`
